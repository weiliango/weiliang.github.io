    Bonferroni's Principle - Meaningfulness of Analytic Answers

Def.

If your method of finding significant items returns significantly more items that you would expect in the actual population, you can assume most of the items you find with it are bogus.

Example:

Suppose we believe that certain groups of evildoers are meeting occasionally in hotels to plot doing evil. We want to find people who at least twice have stayed at the same hotel on the same day.

Details:

  • One billion ($10^9$) people being tracked
  • 1000 days
  • Each person stays in a hotel 1% of the time (10 days out of 1000)
  • Hotels hold 100 people
  • $10^5$ hotels to hold 1% of $10^9$ people
  • If everyone behaves randomly, will the data mining detect anything suspicious?

Solution:

First, we can calculate the percentage of one specific people $p$ and $q$ will be at the same hotel on given day: $$ \frac{1}{10^5} \times \frac{1}{100} \times \frac{1}{100} = 10^{-9} $$ Then, Probability that p and q will be at same hotel on given day $d_1$ and $d_2$ is: $$ 10^{-9}\times10^{-9} = 10^{-18} $$ So, the possible pairs of dates are: $$ \binom{1000}{2} =\frac{1000\times999}{2} \approx 5 \times 10^5 $$ And, the possible pairs of people are: $$ \binom{10^9}{2} \approx 5 \times10^{17} $$ Therefore, number of people who at least twice have stayed at the same hotel on the same day is $$ 10^{-18} \times 5 \times 10^5 \times 5 \times 10^{17} = 25 \times 4 = 250000 $$ In conclusion, the result is meaningless, that's Bonferroni's Principle.

Large-Scale File Systems and Map-Reduce

Large-scale Computing

Challenges:

  1. How do you distribute computation?
  2. How to write distributed programs easily?
  3. How to handle machines fail?

Solution:

  1. Bring computation to data

  2. Store files multiple times for reliability

    $\rightarrow$ MapReduce

MapReduce

Storage Infrastructure

If nodes fail, how to store data persistently?

$\to$ use distributed file system

Usage pattern?

  1. Huge files
  2. Data rarely updated
  3. Reads and appends are common

Distributed File System

  • Chunk servers
    • file is split into ==contiguous== chunks
    • each chunk 64M
    • each chunk replicated 2x or 3x
    • keep replicas in different racks
  • Master node
    • stores metadata about where files are stored
    • known as Name Node
    • might be replicated
  • Client library for file access
    • talks to master to find chunk servers
    • connects directly to chunk servers to access data

Programming Model : MapReduce

3 steps of MapReduce

  1. Map: extract something you care about(keys)

    • Read input and produces a set of key-value pairs

    • details:

      • some number of Map tasks are given one or more chunks from DFS
      • written by user
      • processes chunks and produces key-value pairs
      • normally the keys of input elements are not relevant
      • keys do not have to be unique
    • Combiner

      • when a reduce function is ==associative and commutative==(can be combined in any order with same result), so that we can put some Reduces' work to combiner

      • still required to do aggregation at Reduce task for key

      • reduces the total bandwidth

        • ==About use reduce task as a Combiner== $\infty$

        • if a reduce function is both commutative and associative, it can be used as a combiner as well

        • in some case, reducer cannot be directly used as a combiner because of associativity and commutativity

        • you might still be able to write a third class to use as a combiner for your job

  2. Group by key: sort and shuffle

    • Collect all pairs with same key(Hash merge, Shuffle, Sort, Partition)
    • details:
      • collects key-value pairs from each Map task
      • divides keys among all reduce tasks
      • all key-value pairs with same key go to same Reduce task
      • grouping is performed by the system, regardless of what the Map and Reduce tasks do
        • r Reduce tasks (specified by user)
        • the master controller typically applies a hash function to keys and produces a bucket number from 0 to r-1
        • each key that is output by a Map task is hashed and its key-value pair is put in one of r local files
        • each file is sent to one of the Reduce tasks

  3. Reduce: Aggregate, summarize, filter or transform

    • Collect all values belonging to the key and output
    • details:
      • works on one key at a time
      • reduce code written by user: combines all values associated with that key in some way
      • produces output key-value pairs
