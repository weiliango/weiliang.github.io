Matrix Multiplication

$C = A \times B$

A has dimensions $L\times M$

B has dimensions $M \times N$

C has dimensions $L \times N$

Matrix multiplication: $C_{[i,k]} = \sum_{j}(A_{[i,j]}\times B_{[j,k]})$

  • Map Phase:
    • for each element (i,j) of A, emit $((i,k), i, j, A_{[i,j]})$ for $k \in [1, N]$
    • for each element (j,k) of B, emit $((i,k), j, k, B_{[j,k]})$ for $i \in [1,L]$
  • Reduce Phase:
    • key = (i,k)
    • value = $\sum_{j}(A_{[i,j]} \times B_{[j,k]})$

Two-phase Map/Reduce Matrix Multiply

  • 1st Map Function:
    • for each matrix element $A_{[i,j]}$, emit $((j), A, i, j, A_{[i,j]})$
    • for each matrix element $B_{[j,k]}$, emit $((j), B, j, k, B_{[j,k]})$
  • 1st Reduce Function:
    • for each value of $(i,k)$ which comes from $A$ and $B$, emit $((i,k), A_{i,j} \times B_{j,k})$
  • 2st Map Function:
    • let the pair $((i,k), A_{i,j} \times B_{j,k})$ pass through
  • 2st Reduce Function:
    • emit $((i,k), \sum values)$

More about Map Reduce

Data Flow

  • input and final output are stored on a dfs(hdfs)
  • intermediate results are stored on local FS of Map and Reduce workers
  • output is often input to another MR task

Coordination

  • When a map task completes, it sends the master the ==location and sizes== of its intermediate files, one for each reducer
  • Master pushes this info to reducers
  • Master pings workers periodically to detect failures

Dealing with Failures

  • Map worker failure
    • map tasks completed or in-progress at worker are reset to idle
      • The reason for redoing completed Map tasks is that their output destined for the Reduce tasks resides at that compute node, and is now unavailable to the Reduce tasks
    • reduce workers are notified when task is rescheduled on another worker
  • Reduce worker failure
    • only in-progress tasks are reset to idle
    • reduce task is restarted
  • Master failure
    • MR task is aborted and client is notified

How many Map and Reduce jobs?

  • $M$ map tasks, $R$ reduce tasks
    • make $M$ much larger than the number of nodes in the cluster
    • one DFS chunk per map is common
    • improves dynamic load balancing and speeds up recovery from worker failures
    • do pipeline shuffling with map execution
    • minimizes time fro fault recovery
  • Usually $R$ is smaller than $M$
    • because output is spread across $R$ files
      • means: we need to create intermediate files for each Reduce task, so if there's too large $R$, the intermidiate files explode

Refinements:

Backup Tasks

  • target:
    • to solve the problem of slow workers
  • solution:
    • near end of phase, spawn backup copies of tasks
      • whichever one finishes first "wins"
  • effect:
    • dramatically shortens job completion time

Combiners

  • if a combiner is used then the map key-value pairs are not immediately written to the output, they will be collected in lists, one list per each key value
  • when a certain number of key-value pairs have been written, this buffer is flushed by passing all the values of each key to the combiners's reduce method and outputting the key-value pairs of the combine operation as if they were created by the original map operation.

Definition of commutative and associative

commutative

  • if changing the order of the operands does not change the result

$$ \forall x,y \in S, x * y = y * x $$

associative

$$ (x* y)* z = x*(y * z) $$

example: sum(), max()

Partition Function

control how keys get partitioned

  • inputs to map tasks are created by contiguous splits of input file
  • Reducer needs to ensure that records with the same intermediate key end up at the same worker

default partition function, can be override $$ hash(key) \mod{R} $$

  • Good for applications that work in batch mode

  • suited for data can be extracted as key-value without losing context, dependencies

Cost Measures for MapReduce

  1. Communication cost = total I/O of all processes
    • input file size +
    • sum of the sizes of all files passed from map to reduce +
    • sum of output sizes of reduce
  2. Elapsed communication cost = max of I/O along any path
    • largest input +
      • s could be
        • what fits in memory
        • what fits on local disk
    • output for any map process +
    • output for any reduce process
  3. (Elapsed) computation cost analogous, but count only running time of processes

What Cost Measures Mean

  • Either the communication or computation cost dominates

    • ignore one or the other

  • With proper indexes, computation cost is linear in the (input+output) size

Introduction of SPARK

Apache Spark is an open source cluster computing framework

Application areas:

  • Batch processing
  • Interactive data query
  • Real-time data analysis

SPARK VS. HADOOP

  • Faster(10x on disk, 100x in memory)

    • in-memory caching abstraction

  • ecosystem is simple

Advantages

  • runs everywhere

  • REPL - interactive command line

RDD

RDDs are fault-tolerant, parallel data structures that let users explicityly persist intermediate results in memory, control their partitioning to optimize data placement, and manipulate them using a rich set of operators

two operations in RDD

  • Transformation
    • return new RDD
  • Action
    • return value
    • e.g. first(), count()

Driver Program: SparkContext

  • every application consists of a driver program that launches various parallel operations
  • the driver program contains your app's main function and defines distributed datasets

Important Concepts:

  • worker: the nodes which can run applications in a cluster
  • executer: worker nodes' processes used to running individual tasks
  • job: triggered by action operator, consisting of server tasks
  • task: a work unit which is sent to executor
  • stage: job is divided into several sets of tasks, each set is a stage

diff reduceByKey vs groupByKey

  • same result
  • reduceByKey more efficient, reduces some unnessary shuffle
